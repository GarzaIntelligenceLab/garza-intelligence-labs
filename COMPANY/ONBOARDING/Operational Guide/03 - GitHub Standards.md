# 03 - GitHub Standards

> **Status:** Approved
>
> **Version:** 1.0
>
> **Last Updated:** July 2026

---

## Purpose

Establish the GitHub standards used by Garza Intelligence Labs to ensure consistency, accountability, and long-term maintainability across all repositories.

GitHub is more than a storage platform. It serves as the historical record of Garza Intelligence Labs' engineering, documentation, and product development.

---

## Philosophy

Every meaningful contribution should be documented.

Every meaningful change should be traceable.

Every repository should reflect the Triple O Philosophy through organization, consistency, and continuous improvement.

---

## Repository Standards

Repositories should:

- Serve one primary purpose.
- Follow consistent naming conventions.
- Contain clear documentation.
- Maintain organized folder structures.
- Avoid unnecessary duplication.
- Preserve historical decisions through version control.

---

## Documentation Standards

Every repository should include:

- README.md
- LICENSE (when applicable)
- Relevant Operational Guides
- Relevant Testaments when appropriate
- Markdown documentation following GIL standards

Documentation should always be updated alongside meaningful changes.

---

## Branch Standards

Version 1 Standard:

- Main development occurs on the default branch.

As Garza Intelligence Labs grows, feature branches and pull request workflows will be introduced.

---

## Commit Standards

Every commit should represent one meaningful unit of work.

Commit messages should be concise and descriptive.

Examples:

- Add Product Philosophy Testament
- Complete Executive Candidate Evaluation
- Update Development Environment Guide
- Refine Capability Framework

Avoid commit messages such as:

- Update
- Stuff
- Changes
- Fix

Future team members should understand what changed by reading the commit history.

---

## Repository Organization

Repositories should remain organized through consistent folder structures.

Large files should not be duplicated unnecessarily.

Artifacts, Testaments, Operational Guides, research, and product documentation should remain separated according to established GIL standards.

---

## Security

Never commit:

- Passwords
- API Keys
- Private credentials
- Sensitive customer information
- Proprietary information not intended for the repository

When in doubt, leave it out.

---

## Executive Responsibility

Every executive is responsible for protecting the quality of the repository.

Version control is not simply an engineering practice.

It is the historical record of Garza Intelligence Labs.

---

## Verification Checklist

□ GitHub account authenticated

□ Repository cloned successfully

□ Repository structure verified

□ Documentation updated

□ Commit message follows GIL standards

□ Repository ready for collaboration
